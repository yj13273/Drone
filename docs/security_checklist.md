# UAV Route Optimization Security Checklist

## 1. HTTPS / Transport

- Production uses HTTPS through the reverse proxy
- HTTP redirects to HTTPS through the reverse proxy
- HSTS enabled in production only
- Backend and frontend are not directly public in production
- Local development stays on `http://localhost:5173` and `http://localhost:5000`
- Replace `your-domain.com` in the production Caddy config before deployment

## 2. CSP

- `default-src 'self'`
- `script-src 'self'`
- `object-src 'none'`
- `base-uri 'self'`
- `frame-ancestors 'none'`
- `connect-src` restricted to the API origin / same origin in production
- Avoid `unsafe-inline` for scripts
- `style-src 'unsafe-inline'` allowed only because the current UI uses inline style attributes and style tags generated by trusted frontend code

## 3. XSS

- No `dangerouslySetInnerHTML`
- No `innerHTML` or `outerHTML` writes with user content
- No `eval` / `new Function`
- Backend responses rendered as text by React
- CSV content rendered as plain text
- Uploaded filenames, run IDs, backend errors, drone names, algorithm names, sensor labels, and table cells are treated as plain text

## 4. Clickjacking

- `frame-ancestors 'none'`
- `X-Frame-Options: DENY`

## 5. MIME Sniffing

- `X-Content-Type-Options: nosniff`
- Correct `Content-Type` set by backend and Nginx

## 6. CSV Safety

- `.csv` extension only on frontend upload selection
- Frontend max file size enforced at 5 MB per CSV
- MIME type checked when available, without trusting MIME alone
- Pasted CSV parsed as plain text
- Frontend-generated CSV export escapes spreadsheet formula prefixes
- CSV content is never rendered as HTML

## 7. CORS / API

- No hardcoded `localhost:5000` in production frontend source
- Frontend uses `VITE_API_BASE_URL`
- Production frontend uses same-origin `/api`
- Backend CORS uses an allowlist
- Credentials remain disabled unless explicitly needed later

## 8. File and URL Safety

- `encodeURIComponent` used on dynamic route parts
- `runId` validated against `/^[A-Za-z0-9_-]{1,100}$/`
- Plot and CSV filenames selected from allowlists
- Frontend never constructs filesystem paths
- Backend rejects traversal attempts and serves only validated files

## 9. Dependency Security

- `npm audit` should be checked for the frontend
- High and critical issues should be remediated where safe
- Unused dependencies should be removed where low risk
- Avoid adding heavy dependencies for simple UI logic

## 10. Client Identity

- `clientId` in localStorage is only a local demo identifier
- `clientId` must not be treated as secure user identity
- Real authentication and authorization are required before public deployment
- Backend must enforce object-level access control for runs once user accounts exist

## 11. Backend Hardening

- `X-Powered-By` hidden
- Helmet-based API hardening enabled
- NoSQL operator sanitization enabled
- HPP protection enabled
- Rate limiting enabled on `/api`
- Unknown routes return JSON without leaking filesystem paths

## 12. File Serving Security

- Filename allowlists enforced on backend
- Run ID validation enforced on backend
- Path traversal prevention enforced on backend
- No absolute path leakage
- Attachment disposition set for downloadable CSVs

## 13. Process Execution Security

- No user-controlled shell commands
- `RUN_DIR` constructed server-side
- Worker only runs the fixed pipeline command

## 14. Secrets and Environment

- No secrets committed
- `.env` files ignored
- Example env files committed without secrets
- Production secrets injected through environment

## 15. Database Security

- MongoDB not publicly exposed in production compose
- Least privilege still needed if auth is added later
- Backup / recovery planning still needed for real deployment
